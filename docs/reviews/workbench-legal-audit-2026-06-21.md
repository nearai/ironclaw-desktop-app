# IronClaw Workbench — Legal Daily-Driver Audit

**Scope:** gateway `connector-route-on-main` + desktop `workbench-overnight-20260620`
**Question:** Does this work as a daily chief-of-staff for a LEGAL use case (contract review, drafting, Gmail/Calendar/Notion/Drive, approvals) with the user's own MCP connectors?

---

## 1. Verdict: **YELLOW**

The security spine is sound enough to trust with connected accounts: keys are validated before persist, the SEND gate is fail-closed and exact-match, connected-sources is genuinely read-only to the agent, and no secret leaks to logs or the browser. Nothing here is a RED-grade exploitable hole for a single-user desktop deployment — every confirmed security finding is bounded by the operator/owner trust line. But it is **not GREEN for a legal daily driver** because the workflow surface a cautious legal operator actually needs is missing or unproven: there is **no in-app tool-permission governance** (the one control you want over email-send / calendar-write / file-access during contract work), **no in-app skill management** (the path that would carry contract-review/drafting workflows), connector reads are **default-on with no per-turn consent or opt-in**, and the **draft/send/connector/model round-trip is unproven end-to-end** in a static checkout (model "verified" requires a live run; OAuth needs a live gateway + provider client IDs). It will run, but key guardrails are absent and the legal-workflow happy path is unverified. Close the items in §5 and this moves to GREEN.

## 3. Security Posture: Sound for single-user desktop, with two named soft spots

**Key handling — mostly sound, one real soft spot.** Credentials are validated before persistence (`nearai_login.rs:147` only vaults after `/v1/users/me` succeeds; `nearai_connect.rs:393` mints + vaults `sk-`), never logged, never returned to the browser (`api_key_set: bool` only, `llm_config_service.rs:214`), and injected env-only into the sidecar. The one genuine gap: the NEAR AI credential can be **Bearer-sent to an operator-redirected host** because the host-allowlist / egress-pin the validator comment claims does not exist (`nearai_chat.rs:192`, `reborn_services.rs:793`). Operator-bound, so MEDIUM not HIGH — but it is the single most worthwhile hardening item.

**Write-gate — fail-closed and correct, with a denylist-shaped weakness.** The SEND gate (`IRONCLAW_WORKBENCH_SEND_ENABLED`) is exact-match, case-sensitive, deployment-time, and rejects every near-miss (`connectors.rs:274-283`). The READ gate is the inverse shape — a verb *denylist* — which is the wrong primitive for a security boundary, but it is actively maintained, blocks all canonical destructive verbs, and has no demonstrated real bypass (LOW). The one un-gated mutation is **draft creation** (`connectors.rs:357-382`): a real mailbox write with no approval interstitial, though WebUI-human-only and non-delivering.

**Connector read-only — genuinely enforced, multi-layer.** The agent's only capability is `connected-sources.read` (manifest forbids sends/mutations); the handler routes solely through `ConnectorReadPort.read()` (`connected_sources.rs:63-71`); write-shaped reads are rejected by allowlist; an unconnected account returns `Unavailable` server-side. `GMAIL_SEND_EMAIL` is rejected on both routes. The agent cannot mutate a connector and cannot read an account the user never connected.

**Approval boundaries — the actual weak point for legal.** There is **no approval/checkpoint logic anywhere in the connector write path**, and **no per-turn consent gate on reads** (default `allow`). For most desktop use this is bounded by the human-WebUI-only write route and read-only agent reads. For a *legal* operator who wants an explicit "ask before touching my mailbox/calendar" boundary, this is the missing control — see §4.

Net: **no exploitable escalation, no secret leakage, no agent-reachable mutation.** Every confirmed security finding sits inside the owner/operator trust line. Posture is sound for the single-user desktop it targets; it is *not* hardened for multi-tenant (`reborn_services.rs:3120` discards the caller) and should not be mounted that way without threading per-caller scoping.

## 4. Honest gaps that block "works for my legal use case"

1. **No in-app tool-permission governance** (`settings-api.js:108-113`). A legal operator wants per-tool `always_allow / ask / disabled` over email-send, calendar-write, and file access during contract work. The runtime model exists server-side (`tool_permission_set`, locked-tool enforcement), but the desktop UI is a read-only stub. **Next step:** ship the v2 tools read+write endpoint and wire the editor; until then, the agent is governed only by server-side defaults the user can't see or change.

2. **No in-app skill management** (`settings-api.js:120-128`). Skills are the path that would carry contract-review and drafting workflows, and they cannot be installed/managed in-app. The backend already ships (`ironclaw_webui_v2/src/router.rs:238-252`). **Next step:** this is a ~one-file frontend wiring change — point `fetchSkills/installSkill/removeSkill` at `/api/webchat/v2/skills*` (mirror `fetchExtensions`); remove the stale TODO comments.

3. **Default-on connector reads with no consent** (`sidecar.rs:142`). The moment Gmail/Calendar/Drive/Notion is connected, any agent turn reads it silently — undocumented, no opt-in, no ask-mode. For legal/privileged material this is the wrong default. **Next step:** one-time consent on first connect ("IronClaw can read your connected accounts during tasks"), or gate reads behind `PermissionMode::Ask`; drive the trigger poller from a user setting; document the behavior.

4. **Live OAuth + model-output + draft/send round-trip is unproven** in this static checkout. The login, model-readiness, and connector-OAuth code is real, validated, and correctly guarded (INFO findings) — but end-to-end execution depends on a live v2 gateway + provider client IDs (Google connectors need a configured Desktop client ID, `useExtensions.js:423`). **Next step:** run the sign-off pass — NEAR AI sign-in + one chat run (confirm `model_execution_verified` flips and the badge shows "Execution verified") + one live connector read (Gmail/Calendar) + a draft-create + a gated send-attempt (confirm rejected when flag off).

5. **No pre-run model validity check** (`model-readiness.js:67`). A stale/renamed/decommissioned configured model id passes both preflight gates and fails on the first turn with a generic error. **Next step:** map the model-not-found 400 to an actionable "pick another in Settings/Inference" message (cheap) or add an availability probe to the readiness gate (robust).

6. **Multi-step convergence is untested.** No hermetic coverage exists for the multi-tool legal workflow (read contract → draft reply → request approval → send). The three E2E scripts are live-LLM-flaky and not CI-grade (`trigger-fire-e2e.mjs:110`), and the write-gate × classification logic has no port-boundary unit test. **Next step:** add the hermetic write-gate unit tests (§2) so convergence behavior is regression-locked, and keep the live scripts as manual/nightly smoke only.

## 5. Prioritized remediation (top 5, most impactful first)

1. **Pin the NEAR AI credential to a host allowlist** (`llm_config_service.rs:274`, `nearai_chat.rs:192`, `reborn_services.rs:793`). Gate env-key/session-token attachment on the *resolved* host matching `cloud-api.near.ai`/`private.near.ai` + explicit `NEARAI_BASE_URL`, independent of the operator-mutable persisted endpoint; route probe clients through a real egress guard (resolve → re-validate → pin IP). Closes the only MEDIUM security finding and its two INFO/LOW satellites.

2. **Wire the skills UI to the existing v2 backend** (`settings-api.js:120-128` → `/api/webchat/v2/skills*`). Highest leverage for the legal use case at the lowest cost — the backend already ships; this is a frontend wiring change that unlocks contract-review/drafting skill management.

3. **Ship in-app tool-permission governance** (`settings-api.js:108-113`). Build the v2 tools read+write endpoint and the editor so the operator can set `ask/allow/disabled` over email-send / calendar-write / file-access. This is the guardrail a cautious legal operator most needs.

4. **Add connector read consent + a model-not-found error path.** Surface a one-time consent on first connector connect (or `PermissionMode::Ask`) and document the default-on behavior (`sidecar.rs:142`); map the runtime model-not-found 400 to an actionable Settings/Inference message (`useWorkbenchStart.js:37`, `model-readiness.js:67`). Two small changes that remove the worst surprises.

5. **Lock the write-gate with hermetic tests + assert sidecar env wiring.** Extract the gate decision and sidecar env into pure helpers; unit-test all 4 write kinds × `send_enabled`, the traversal guard, secret validation, and the presence of both sidecar env vars (`connectors.rs:357`, `sidecar.rs:142`). Converts the currently-correct-but-untested security boundary into a regression-locked one.

**Sign-off blocker before calling it GREEN:** run the live round-trip in §4 item 4 (NEAR AI sign-in + chat run + one connector read + draft-create + gated-send-reject). The code is real; only live execution is unproven.
