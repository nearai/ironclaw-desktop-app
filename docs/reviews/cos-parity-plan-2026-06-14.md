# IronClaw Desktop — Agentic Chief-of-Staff Parity Review & Build Plan (2026-06-14)
Escalating CoS product review + feature-parity gap analysis, grounded in the real reborn gateway capability map, adversarially verified. 30 agents; 17 proposals kept, 2 rejected.
## Verdict

IronClaw Desktop sits at the FRONT of the personal-chief-of-staff field on the two scarcest axes (action-receipts/approval-gating and exportable work-product) and at parity-MINUS on the two axes peers treat as baseline (proactive "what-moved" sensing and durable cross-matter findability/memory). Verified against the real shipping UI — the static React+htm WebUI under crates/ironclaw_webui_v2_static/static/js, NOT the legacy SvelteKit src/ tree (README confirms this explicitly; the proposals are correctly written against the static layer). The maturity curve is inverted vs. the category: IronClaw already ships engineering-grade approval choreography (approval-card.js + approval-risk.js: action/touches/destination/what-leaves-machine + risk badge + over-reports cc/bcc as outbound), collapsed Handled receipts with gold attribution (tool-activity.js), client-side full extraction of PDF/OCR/DOCX/XLSX/PPTX so the model actually reads files, and hand-built OOXML + mermaid-rasterized MD/DOCX/PDF/HTML/JSON export from both chat and Work page — none of which Cora, Martin, alfred_, Lindy or Xembly match. The three capabilities that most define the remaining gap: (1) cross-thread findability — search is title-only and client-side (sidebar-threads.js:173 filters thread.title||thread.id; GlobalSearch fans across 6 surfaces but Threads is a client cache of listThreads, no message-body route exists), AND the thread list never paginates (useThreads.js still calls listThreads({}) and exposes next_cursor at line 64 that nothing consumes — the June-14 batch-C fix added loading/error/retry states but did NOT wire the cursor), so old work is literally unreachable; (2) proactive sensing — the Morning Brief is derived only from local thread state, has no last-seen watermark, and the automations/routines viewers are hidden:true (routes.js:12-13) with no nav path; (3) durable matter memory — work-product-save.js reserves dossier/approvalBoundaries/receipts/followUps and writes them empty, persistence is a 500-cap localStorage blob, and there is no server audit/search/memory route. IronClaw's frontier is NOT to copy Martin's phone calls or Cora's twice-daily Brief; it is to compound its receipt+approval+export foundation into a queryable trust ledger and durable matter dossiers — the exact white-space (research DIMs 11/12) that no peer covers — while fixing one live correctness defect (fetchRunState GET /runs/{runId} 404s, api.js:477).

## Escalating review (tier 1 → frontier)

TIER 1 — TABLE STAKES (where it's solid, what's broken-by-omission). IronClaw clears most of Tier 1 with quality that exceeds the personal-CoS field. Genuinely done well, do not re-propose: chat send, thread create/list, SSE+WebSocket streaming with Last-Event-ID resume, cursor-paginated thread RELOAD at 50/page preserving optimistic rows (useHistory), and the strongest attachments-in story in the category (client-side PDF/OCR/DOCX/XLSX/PPTX extraction shipped as text/plain so the model actually reads files, with per-chip 'Model can read this file' honesty). Three real basics are still open: search is title-only and client-side (sidebar-threads.js:173); the thread LIST never paginates (useThreads still calls listThreads({}); the June-14 batch-C fix added honest loading/error/retry branches but did NOT consume next_cursor at line 64, so old threads are unreachable by sidebar, search, and palette); and fetchRunState polls an unregistered GET /runs/{runId} that 404s on every call (api.js:477) — the safety net under streaming is dead. There is also no Cmd+F find-within-thread. None of these require inventing readiness: pagination and the bug are pure gateway-backed/client fixes, in-thread find is client-only, and true server search is the one honest upstream ask.

TIER 2 — COS BASICS (what's missing). IronClaw already beats the field on the two thinnest dimensions — receipts/audit-of-actions and structured gating — via the approval gate (action/touches/destination/what-leaves-machine + risk badge, refuses to infer missing fields, over-reports cc/bcc as outbound) and collapsed Handled receipts with gold attribution. It lags only on proactive sensing and durable matter memory, and the gaps are structural: the front door lives inside the chat empty-state rather than a standing surface, the Morning Brief is derived only from local thread state with no last-seen watermark (a connector run or completed automation while away does not surface), the approval inbox is in-thread only (chat.js explicitly defers cross-thread gate visibility — the single biggest 'needs my decision' miss for a parallel-work CoS), the Work surface is the artifacts slice only (work-product-save.js reserves dossier/approvalBoundaries/receipts/followUps and writes them empty), saved work is a 500-cap localStorage blob, and the server-side summary_artifacts the timeline returns has zero readers. The buildable answers: a server-sensed brief, a cross-thread Needs-You inbox, and the durable Matter Dossier that fills the already-reserved fields.

TIER 3 — HARD AGENTIC (the inverted shape). IronClaw is strong on the trust-receipt/approval half and weak on the proactivity/durability half — the inverse of every credible peer, which ship digests with no receipts behind them. Proactivity: the automations viewer is fully built and honest (gold attribution, not fake live-success) but hidden:true (routes.js:13) with no nav path, and listAutomations is read-only — so a user cannot even SEE what fires while away, let alone schedule from the UI. Durable memory, a queryable audit/receipts read-API, follow-ups/reminders, and undo are all absent (no routes). Cross-matter dossiers are partial in a telling way: the schema anticipates the full ask/plan/approvals/receipts/next-steps shape but the UI ships only artifacts. Highest-leverage moves: (a) light up the honest read-only Scheduled viewer that already exists behind the flag; (b) close trigger-creation honestly (upstream-gated — needs a gateway route since agent-tool-only creation is the real constraint); (c) promote the reserved dossier fields into a real durable-matter ledger fed by data the UI already holds (timeline receipts + approvals + artifacts) — the audit-ledger + dossier white-space no peer covers, without faking anything.

TIER 4 — FRONTIER (what would make it best-in-class). IronClaw already holds the two scarcest assets in the category (DIM 11 receipts/audit, DIM 12 sourced dossiers) — engineering-grade approval gates, a client-side risk classifier gating 'Always allow', gold-attributed handled receipts, and mermaid-rasterized multi-format export that only Devin and Sierra approach, and they are eng/CX agents not a personal desk. So the frontier is NOT to copy Martin's phone calls or Cora's twice-daily Brief (table-stakes, mostly gateway-gated here); it is to compound the receipt+approval+export foundation into earned overnight trust: a brief that ranks what moved since last open with every row backed by a linked receipt, a research dossier that is a durable matter with sourced claims rather than a chat trace, and a queryable Trust Ledger of every action. The honest ceiling is hard-capped exactly where peers differentiate — trigger CRUD is agent-tool-only, outbound has zero default providers, durable memory has no route, and there is no audit-read API despite every route declaring an AuditTraceClass. So Tier 4 is two-track: compound the receipt/dossier/trust assets that CAN be proven into best-in-class device-scoped surfaces now, and file the four gateway gaps as honest stubs that light up when the endpoints land — plus fix the fetchRunState 404. Anything implying overnight autonomous action before those land is fake readiness and is forbidden.

## Buildable now (gateway-backed)

- **Capability:** Reliable run-completion detection — the fallback under SSE for the core chat loop. fetchRunState (api.js:477) GETs /api/webchat/v2/threads/{tid}/runs/{rid}, which is NOT a registered route (only .../cancel and .../gates/{ref}/resolve exist). It is polled in useChat.js:333-388 inside try/catch, so it 404s on every poll and the safety net under streaming is dead.
- **UI:** Replace the pollRunState body so run completion/failure is derived from the registered GET .../timeline (which already carries run/message records and is consumed by useHistory) instead of the unregistered bare GET. Reuse the existing timeline parser to read terminal status. No user-facing surface changes; the thread simply leaves 'processing' correctly when SSE drops. Keep the try/catch but stop issuing a request guaranteed to 404 so logs/network stay honest.
- **Acceptance:** No client code path issues GET .../runs/{runId} (bare); completing a turn with SSE artificially dropped still transitions the thread out of 'processing' via the timeline-derived fallback within one poll interval; failed runs still surface the failure message; api.test.mjs gains a regression assertion that no client fetch targets an unregistered route; existing chat tests pass.
