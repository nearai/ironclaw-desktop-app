## 1. Architecture detail

The original "drop react-router, render `<WorkbenchPage/>` directly" is wrong and is the single biggest underestimated coupling. The page and 5 components navigate to sibling routes the standalone would delete, turning start-work and every "Open in…/Set up source/Settings" action into dead links. The standalone therefore keeps `BrowserRouter` and makes an explicit per-target decision (carry `/chat`; stub the rest).

The Workbench tree itself has **zero direct Tauri coupling** (no `tauriInvoke`/`__TAURI__`/`isDesktopRuntime` under `pages/workbench/**`), so the shell scaffold is the carry-over cost, not the page.

**Auth model — verbatim, two bearers ride `Authorization: Bearer`:**
1. Local default: per-install UUIDv4 from `keychain::get_or_create_local_token`, injected as `IRONCLAW_REBORN_WEBUI_TOKEN`/`GATEWAY_AUTH_TOKEN` (`sidecar.rs:128,130`), read back by `bootstrapDesktopSession()` (`api.js:243-287`). Browser holds only this gateway bearer.
2. Provider creds (NEAR sk-/session, OpenAI, Anthropic) stay sidecar-env-only. Preserve the routing invariant: sk- → `cloud-api.near.ai`; NEP-413 session → `private.near.ai`; never both (`sidecar.rs:245-261`). Preserve the boot-abort guard (`NEARAI_BASE_URL` without a key crash-loops) and the `inherited_sidecar_env` allowlist that strips ambient `ANTHROPIC_*/OPENAI_*/NEARAI_*/LLM_*` (`sidecar.rs:435-457`).

**Tauri shell carry-over (Phase 5):** `sidecar.rs` (spawn/health-poll/kill/port-pick/env), `lib.rs` `gateway_http_fetch` + SSRF allowlist + `tauri-plugin-http` (the CORS-bypass seam: webview origin `tauri://localhost` is not in gateway CORS), `keychain.rs` with file fallback, `nearai_connect.rs` + `nearai_login.rs`, window/CSP/capabilities (loopback CSP + `shell:allow-execute` on the sidecar).

## 2. Extract vs share vs re-implement

**Decision: keep ONE tree for v1. Do NOT do the `packages/ironclaw-webui-shared` git-subtree extraction.** `serve-webui-static.mjs` already resolves every relative import across the single tree and proxies the gateway — no package boundary is needed to ship. A subtree adds `git subtree pull` ceremony on every gateway bump and the `api.js`-drift risk for zero Phase-1 benefit. Defer any package split until a *second* consumer actually exists and the two trees have measurably diverged.

| Concern | Decision | Why |
|---|---|---|
| Whole frontend tree (`lib/`, `pages/workbench`, `pages/chat`, `design-system/{icons,theme,modal}`, `utils/cn.js`) | **one tree, copy in** | Bundler resolves relative imports; no module-resolution or publish step. `button/card/badge`/extra tokens are unused by workbench — drop. |
| `lib/i18n.js` + `I18nProvider` + an `en` bundle | **REQUIRED, not optional** | The shared chat closure hard-imports `useT` in 15+ components (message-list, chat-input, approval-card, attachment-preview, connection-status, tool-activity, auth-*-card…). `useChat(null)`/`useComposerAttachments` drag these in transitively. Without `I18nProvider` + a real `en` bundle, shared chat components throw or render raw keys. |
| Chat engine: `useChat` (806L) + `useComposerAttachments` + `useChatEvents` + `lib/history-messages` (860L) + ~22 transitive modules | **carry whole** | Co-owned 1600+ line send/stream engine. `useChat` imports ONLY `lib/`+`pages/chat/lib` modules — **no** `useOutletContext`/`useNavigate`/`useSidebar`/`gateway-layout`. It extracts cleanly (this is verified; see §6). |
| `lib/query-client.js` singleton | **carry; mount as ancestor** | Page calls `useThreadStates()`/`useThreadAttentionDetails()`/`useExtensions()` directly — all need the QueryClient. |
| Outlet context (5 fields) | **supply via `<Outlet context=…>`** | `workbench-page.js:318` is `useOutletContext() || {}` — null-safe. No source edit needed in Phase 1; mount it via a one-route Outlet and pass all 5 fields (`gatewayStatus`, `gatewayStatusQuery`, `currentUser`, `isAdmin`, `threadsState`). |

The original "one required edit at line 318" claim is wrong on two counts: the line is already null-safe, and patching it would not address the three direct hooks (321-323) that actually gate Home's data. The real requirement is provider-order + Outlet context, not a source edit.

## 6. Riskiest assumption / what could make this wrong

**The corrected riskiest assumption is NOT "the chat engine extracts cleanly" — that is disproven** (grep confirms `useChat` imports only `lib/`+`pages/chat/lib`, no shell/router/outlet coupling; `runtime-context.js` is a plain builder, `chat-page.js` wraps `useChat(null)` in no provider; and the existing standalone already runs the live page with `useChat` against a real sidecar).

**The genuinely riskiest, under-flagged assumption is that a distinct identifier is free.** The existing breakout reuses the desktop's keychain entry (`security find-generic-password -s com.openclaw.ironclaw-desktop -a llm-nearai:default`) to get the NEAR credential for free. A new identifier means a **fresh, empty keychain namespace** — the standalone cannot read the desktop's vaulted NEAR/provider creds and must run a full native NEAR login (loopback or in-app WebviewWindow) on first launch before it can talk to NEAR AI Cloud. "Renders against the real gateway" silently becomes "renders only after a fresh OAuth round-trip the web standalone sidesteps entirely." **This is the decision to make explicitly in Phase 3:** either (a) accept and document the fresh-login cost under the new identifier, or (b) read the desktop's keychain entry for first-run credential bootstrap and document the cross-app coupling. Distinct identifier is still right for keyring/updater isolation — but the auth cost must be a conscious choice, not a buried consequence.

**Other risks:** (a) **router** — the audited per-target decision (§(b)) is the gating dependency; any missed `navigate()`/`to=` emitter becomes a dead link. (b) **DESIGN.md conflict** — Phase 2 violates "follow DESIGN.md exactly" without sign-off. (c) **i18n** — missing `en` bundle or `I18nProvider` breaks all shared chat components; the live-render smoke catches it. (d) **chat styling surface** — full content globs required or the send/stream UI renders unstyled. (e) **two sidecars** if both apps installed — different free ports (no collision), double RAM/CPU; document it. (f) **stale budget** — re-baseline desktop `static-bundle-budget.json` after route removal.
