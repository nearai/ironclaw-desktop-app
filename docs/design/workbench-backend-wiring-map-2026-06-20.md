### Route-enabled connector bridge — dev-proven and consumed by Workbench

A parallel Reborn build has proven a read-only Composio connector route:
`GET /connectors/connected` and `POST /connectors/read` return real
Gmail/Calendar/Drive/Docs/Notion/Slack/GitHub availability and read data through
the proxy described in `workbench-mcp-delivery-2026-06-20.md`. The desktop
frontend adapter in `lib/api.js` (`connectorsConnected`, `connectorRead`) is now
consumed by Workbench through `hooks/useWorkbenchConnectors.js` and
`lib/workbench-connectors.js`.

Workbench renders only successful, real connector rows:

- `SourceReadinessStrip` shows ready families only when `/connectors/connected`
  reports an ACTIVE account.
- `WorkbenchArrived` reads `GMAIL_FETCH_EMAILS` and hides or degrades honestly
  on no Gmail, empty inbox, or read failure.
- `WorkbenchUpcoming` reads `GOOGLECALENDAR_EVENTS_LIST` and hides on no
  Calendar, empty calendar, or read failure.
- `WorkbenchSlackBlockers` reads `SLACK_SEARCH_MESSAGES` only after an ACTIVE
  Slack family is present, renders real message text/permalinks, labels the
  result as read-only, never starts Chat or posts a message, and shows an
  explicit Slack search error rather than an invented all-clear when the read
  fails.
- Deterministic catch-up briefings read `GITHUB_LIST_NOTIFICATIONS_FOR_THE_AUTHENTICATED_USER`,
  `GOOGLEDRIVE_LIST_FILES`, and `NOTION_SEARCH_NOTION_PAGE` when those families
  are ACTIVE, render the real rows/links through `WorkbenchBriefing`, label the
  result as read-only, and do not start Chat. If a ready source cannot be read,
  the briefing carries a `sourceProblems[]` row and renders `Could not read`
  instead of a false all-clear.
- Loop 28 adds one narrow gated-write path from the Gmail reading panel:
  `WorkbenchApprove` creates a reviewable `GMAIL_CREATE_EMAIL_DRAFT` through
  `POST /connectors/write`. The modal says `Gmail · create draft (no send)`,
  validates editable recipient/subject/body fields, reports the created draft id
  when present, and repeats that nothing was sent. The client fast-fails
  off-allowlist write tools before network; the gateway remains the authority for
  the write allowlist and any future send capability.

Loop 23 also fixed the client read-only mirror so suffix-verb tools such as
`GOOGLECALENDAR_EVENTS_LIST`, plus `FIND`/`READ`, are allowed, while mixed or
mutating slugs such as `GMAIL_LIST_AND_DELETE_EMAILS` and
`GOOGLECALENDAR_EVENTS_INSERT` reject before any network call. Packaging caveat:
the shipped desktop sidecar still needs the official route-enabled Reborn build;
without it, Workbench must keep these connector-read surfaces hidden or in their
honest error states.

## 3. Required future endpoints (and the exact adapter that should consume each)

Ordered by Workbench impact. "Adapter" names the frontend module that should own
the call (the `file` to set on each finding).

### G1. Saved-Work read path (durable, cross-device)

- Need: server-backed read of saved work items + artifacts, so Library/dock/packet
  show work created on any device or by background runs — not just this browser.
- Today: `readSavedWorkSnapshot()` is localStorage only
  (`work-product-save.js:230-245`) and reports local/unavailable/corrupt source
  state. Chat→Work save is also localStorage
  (`saveAssistantResponseToWork:115-116`).
- Candidate existing surface: per-thread project files already exist
  (`GET /threads/{id}/files`, `lib/project-files-api.js:17`) and `/projects/*`
  (`projects-api.js`) — but there is **no "list my saved work products"** route.
- Future endpoint (does not exist): `GET /work` (+ `GET /work/{id}` / artifact bytes).
- Adapter to add/extend: `crates/.../pages/chat/lib/work-product-save.js`
  (add a `fetchSavedWorkItems()` server reader beside `readSavedWorkItems`),
  consumed by `workbench-page.js:277` and `work-page.js:302`.

### G2. Open approvals feed (cross-thread)

- Need: list of prepared external actions awaiting approval across all threads, to
  populate the dock "Needs approval" group honestly.
- Today: approvals exist primarily as live in-thread gate events (`useChat.js:554`,
  `smoke-gate-enforcement.mjs:316`). Loop 14 added `thread-attention-details.js`,
  so the active browser can preserve a gate headline/body/provider and Workbench can
  render the exact waiting decision. Loops 17-18 added a `/threads` row adapter in
  `workbench-state.js`, so backend-provided `attention_detail`, `pending_gate`,
  or `pending_gates[]` can render the same specific decision without relying on
  browser-local storage, and rows with those fields are promoted into `Needs a
decision` even if the coarse thread `state` is missing or still `running`.
  Backend waiting labels such as `needs_approval`, `awaiting_approval`,
  `auth_required`, and `waiting_for_input` also normalize to needs-attention.
  `savedWorkRows` still reads a local `openApprovals[]` that is never populated
  (`work-product-save.js:110`), and there is no gateway route listing all
  pending gates.
- Future endpoint (does not exist): `GET /approvals` (pending gates across threads)
  with resolve links/status, or a documented `/threads` contract that includes
  `pending_gates[]` for all waiting threads.
- Adapter to add/finish: if using the existing threads route, document and back
  `pending_gates[]`; if adding a dedicated feed, add `lib/approvals-feed-api.js`
  consumed by `pages/workbench/lib/workbench-state.js`.

### G3. Memory / preference proposal + durable storage

- Need: write a scoped preference (Personal/Workspace/Project/Source) and read it back.
- Today: `workbench-memory.js` is a disabled mock (`:54`); `settings-api.js`
  settings writers are TODO stubs (`updateSetting:13`, `importSettings:16`).
- Future endpoints (do not exist): `GET/POST /preferences` (or `/memory`) with scope.
- Adapter to add: `settings/lib/settings-api.js` (replace the `updateSetting` TODO
  stub family), consumed by `pages/workbench/components/workbench-memory.js`.

### G4. Monitor / automation CRUD (create + toggle + delete)

- Need: turn "Watch this weekly" / cadence into a real recurring automation under
  approval. Read is wired (`/automations`); **writes are not.**
- Today: `routines-api.js` write fns reject by design —
  `triggerRoutine/toggleRoutine/deleteRoutine` `:34-46`; cadence is prompt-only
  (`workbench-page.js:252-256`).
- Future endpoints (do not exist as consumed routes): `POST /automations`,
  `POST /automations/{id}/toggle`, `DELETE /automations/{id}`,
  `POST /automations/{id}/run`. (`automations/runs` exists read-only — probe treats
  it as informational/non-gating `probe...acceptance.mjs:559-564`.)
- Adapter to add: extend `crates/.../lib/automations-api.js` (currently read-only
  `listAutomations`), consumed by `routines/lib/routines-api.js:34-46` and a new
  Workbench cadence→automation action in `workbench-page.js` CadenceInspector.

### G5. Receipts / audit / provenance feed

- Need: durable record of completed external actions (what was sent/posted/filed),
  for the dock "Recent receipts" group and per-work provenance.
- Today: receipts are derived locally from tool-activity transcript rows at save time
  (`dossierFromMessages` `work-product-save.js:15-35`), and `receipts[]` is `[]` for
  most saved items (`:87`). No server audit feed.
- Future endpoint (does not exist): `GET /receipts` (or `/audit`), ideally tied to
  resolved gates.
- Adapter to add: a new `lib/receipts-api.js` consumed by `workbench-state.js:164`
  (`savedWorkRows` receipts branch).

### G6. Global pending-work / source-changed feed

- Need: a single "what needs me / what changed while away" stream powering the home
  triage and the `What needs me today?` / `Summarize what changed.` suggestions
  (`workbench-plan.js:62-72`).
- Today: the rail is assembled client-side from threads + local saved items +
  source readiness (`buildWorkbenchStateRail` `workbench-state.js:210-235`). There is
  no server "since last seen" / changed-sources feed.
- Future endpoint (does not exist): `GET /inbox` or `GET /pending` (cross-source
  deltas). The route-enabled connector bridge may become one read input to that
  feed, but only after Workbench has a real adapter/model for connector rows and
  offline/error states.
- Adapter to add: a new `pages/workbench/lib/workbench-feed-api.js` consumed by
  `workbench-page.js:323` (`railGroups` memo) alongside `buildWorkbenchStateRail`.

## 6. Top gaps, ranked

1. **G1 Saved-Work read path** — highest leverage. Library, dock, and packet now
   depend on the `readSavedWorkSnapshot()` local profile boundary, but the data
   still comes from localStorage. Without a `GET /work` reader, the most
   prominent "real artifacts" surface only reflects this desktop profile.
   Adapter: `chat/lib/work-product-save.js`.
2. **G2 Cross-thread approvals feed** — the dock's "Needs a decision" group is the
   product's core promise (approval-gated external actions). Loop 14 makes live
   gates specific in the current browser; Loops 17-18 let `/threads` rows
   carrying `pending_gate`/`pending_gates[]` render specific decisions and be
   grouped correctly even when the coarse `state` is absent or stale. The product
   still needs either a documented, durable thread-list pending-gate contract or
   `GET /approvals` with resolve/status metadata for cross-device,
   never-opened-thread visibility. Adapter: `workbench-state.js` plus, if
   dedicated, new `lib/approvals-feed-api.js`.
3. **G4 Monitor/automation CRUD** — "Watch this weekly" is a first-screen suggestion
   (`workbench-plan.js:107`) but cadence is prompt-only; writes rejected by design.
   Needs `POST /automations` (+toggle/delete/run). Adapter: extend
   `lib/automations-api.js`.
4. **G5 Receipts/audit feed** — "Recent receipts" dock group + provenance need a
   durable server record of completed external actions, ideally keyed off resolved
   gates. Adapter: new `lib/receipts-api.js`.
5. **G3 Memory/preference storage** — Memory tab is an honest mock; needs
   `GET/POST /preferences` with scope. Adapter: `settings/lib/settings-api.js`
   (replace TODO stubs).
6. **G6 Global pending/changed feed** — quality-of-life; `What needs me today?` /
   `Summarize what changed.` currently synthesize from threads + local items, not a
   server delta. Adapter: new `pages/workbench/lib/workbench-feed-api.js`.

No Workbench code should add an optimistic endpoint for any of the above. Each gap
should render staged/empty/unavailable (as Memory and Cadence already do) until the
backing route ships and the named adapter consumes it.

## 7. Packaging / live-preview caveat

The generated static bundle and route tests render `/v2/workbench` correctly. A
temporary preview proxy also rendered that static bundle with `/api` forwarded to
the live local sidecar, proving the current UI can display real NEAR AI model
data (`GLM 5.1 FP8`) and connector readiness instead of mocked catalog data.

However, direct `src-tauri/binaries/ironclaw-reborn-aarch64-apple-darwin serve`
from the currently checked-in prebuilt sidecar redirected `/v2/workbench` to
`/v2/chat`. That binary was built Jun 19 00:06 and appears to carry stale
embedded serve-mode UI. Treat this as a release/readiness caveat, not as proof
that the generated Tauri static root is broken. Before declaring the Workbench
overhaul end-to-end usable in the desktop app, rebuild or refresh the packaged
sidecar/static preview path and repeat:

```bash
npm run prepare:webui-static
npm run build:reborn-sidecars
node scripts/probe-workbench-live-wiring.mjs --probe-oauth-start --json
```

Then open `/v2/workbench` against the rebuilt local app path without a temporary
proxy and confirm it no longer redirects to Chat.
